# ONDAY Catalog - план разработки

## Подход

Проект нужно делать поэтапно, с отдельной проверкой и коммитом после каждого завершенного блока. Нельзя начинать весь проект одной большой командой. Перед кодом нужно согласовать оставшиеся вопросы из `QUESTIONS.md`.

## Этап 0. Согласование

Цель: подтвердить границы первой версии и модели данных.

Уже решено:

- главный визуальный референс - утвержденный темный макет ONDAY Catalog;
- стартовый каталог создается первым полным импортом WhatsApp-прайсов;
- Google Таблица не является источником актуального наличия;
- первая версия парсера оптимизируется для смартфонов;
- архитектура поддерживает все категории;
- стартовые поставщики и глобальный приоритет определены;
- суммы хранятся целыми тенге;
- EAC хранится как `да / нет / не определено`.
- архив через 30 дней скрывается через soft-delete;
- исходные тексты импорта хранятся 12 месяцев;
- вход разрешен по логину или email;
- 2FA предусматривается для владельца и администраторов;
- Telegram в первой версии отправляет уведомления владельцу в личный чат;
- Telegram token хранится только в environment variables;
- хостинг: Vercel + Supabase;
- окружения: `development`, `production`, Vercel preview deployments.
- парсер первой версии работает по правилам, справочникам и алиасам без платного AI API;
- слова про верификацию полностью удаляются при разборе текста;
- продавцу нельзя передавать закупочные цены и настоящие названия поставщиков даже в скрытых API-полях.

Осталось:

- собрать реальные примеры прайсов Дархана и Талисмана;
- перед стартом реализации подтвердить, что дополнительных бизнес-исключений нет.

## Этап 7. Парсер и предварительная проверка

Цель: превратить WhatsApp-текст в управляемый импорт.

Работы:

- создать страницу "Добавить прайс-лист";
- реализовать выбор поставщика, режим, текстовое поле и массовую маржу;
- реализовать parser pipeline для смартфонов;
- не подключать платный AI API для распознавания прайсов в первой версии;
- подключить справочники и aliases;
- полностью удалять слова и строки верификации;
- распознавать категорию, бренд, модель, память, цвет, SIM, EAC и цену;
- разделять несколько цветов на отдельные строки;
- блокировать строки с нераспознанным цветом;
- показывать конфликт, если одинаковый товар в одном прайсе указан с разной ценой;
- показывать preview-таблицу;
- разрешить ручное исправление, удаление и добавление строк.

Проверка:

- пример `17 pro 256gb orange silver blue / 635.000tg` превращается в три строки;
- парсер работает воспроизводимо на правилах, справочниках и алиасах;
- EAC распознается как `yes`, `no` или `unknown`;
- нераспознанный цвет блокирует сохранение строки;
- конфликт цен требует ручного выбора;
- нераспознанные строки не сохраняются автоматически.

- добавить восстановление владельцем;

Проверка:

- карточка товара остается активной при наличии другого поставщика;
- архив скрывается по правилу 30 дней без физического удаления истории;
- исходный текст импорта удаляется только после 12 месяцев, итог импорта остается.

## Тестовая стратегия

- Unit tests: parser, normalization, pricing, supplier priority.
- Fixture tests: реальные форматы прайсов поставщиков.
- Integration tests: import confirmation, permissions, orders.
- E2E tests: login, catalog, import preview, order flow.
- Security checks: role-based API responses, RLS, seller network payload.
- Visual/mobile checks: desktop, tablet, smartphone.
