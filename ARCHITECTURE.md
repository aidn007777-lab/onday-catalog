# ONDAY Catalog - архитектура

## Рекомендуемый стек

Базовая архитектура из ТЗ остается основной:

- Frontend: Next.js, React, TypeScript, Tailwind CSS.
- Backend: Next.js Server Actions и API Routes.
- Database: PostgreSQL через Supabase.
- Auth: Supabase Auth.
- Storage: Supabase Storage для фотографий товаров и логотипов.
- Hosting: Vercel для приложения, Supabase для БД/Auth/Storage.
- Notifications: Telegram Bot API.
- Repository: GitHub.

Такой стек закрывает малобюджетный старт, дает облачную БД, авторизацию, хранилище файлов и деплой без зависимости от ноутбука владельца.

## Общая схема

```text
Browser
  |
  | HTTPS
  v
Next.js App
  |
  | Server Actions / API Routes
  v
Application Services
  |
  | typed queries / RPC
  v
Supabase PostgreSQL
  |
  +-- Supabase Auth
  +-- Supabase Storage
  +-- scheduled jobs / cron
  +-- Telegram Bot API
```

## Принцип разделения данных

Главное архитектурное правило: данные для продавца формируются на сервере отдельными запросами и DTO. Нельзя отправлять продавцу закупочные цены, настоящих поставщиков, маржу, внутренние комментарии и альтернативные предложения даже если UI их не показывает.

Это правило распространяется на все API-ответы, server actions, exports и realtime/payload-события: для продавца запрещены скрытые поля с закупочными ценами и настоящими названиями поставщиков.

Практически это означает:

- отдельные серверные функции для owner/admin и seller views;
- проверку прав на сервере перед каждым действием;
- RLS-политики в Supabase как дополнительный слой защиты;
- отдельные SQL view или RPC для публичного каталога продавца;
- отсутствие прямого чтения чувствительных таблиц с клиентской стороны;
- проверки network payload в тестах для роли seller.

## Слои приложения

### Presentation layer

Next.js App Router:

- страницы входа;
- рабочий layout по утвержденному темному макету;
- каталог в режимах table/cards;
- фильтры каталога;
- правая карточка товара на desktop;
- полноэкранная карточка на mobile;
- импорт прайса и preview;
- заказы;
- настройки, сотрудники, поставщики, аудит.

Desktop-компоновка: левая навигация, центральная таблица, правая карточка. Mobile-компоновка: нижнее или выезжающее меню, карточки товаров, компактные фильтры и полноэкранные детали.

### Server layer

Server Actions и API Routes:

- авторизация и текущий профиль;
- проверка permissions;
- загрузка и подтверждение прайсов;
- чтение каталога по роли;
- создание заказов;
- изменение цен;
- управление сотрудниками;
- отправка Telegram-уведомлений;
- экспорт Excel.

### Domain services

Рекомендуемые сервисы:

- `authService` - текущий пользователь, роль, разрешения, блокировки.
- `permissionService` - централизованные проверки прав.
- `catalogService` - выдача каталога, фильтры, сортировка, карточки.
- `supplierService` - поставщики, склады, глобальные приоритеты.
- `importService` - жизненный цикл импорта.
- `parserService` - разбор текста WhatsApp.
- `normalizationService` - справочники, алиасы, опечатки.
- `deduplicationService` - уникальность товаров и предложения поставщиков.
- `pricingService` - наличная цена, маржа, банковские проценты.
- `notificationService` - системные уведомления и Telegram.
- `orderService` - заявки и статусы.
- `auditService` - журнал действий.
- `archiveService` - нет в наличии, восстановление, очистка через 30 дней.

## Парсер WhatsApp-прайсов

Первую версию парсера оптимизировать для смартфонов, но не зашивать логику только под iPhone. Категория должна определяться через справочники и правила, чтобы позже расширить распознавание планшетов, ноутбуков, часов, наушников, колонок и аксессуаров.

Парсер первой версии должен быть rule-based: правила, справочники, aliases, регулярные выражения и тестовые fixtures. Платный AI API для распознавания прайсов в первой версии не используется.

Pipeline:

1. Очистка текста: пробелы, валюты, технический мусор, полное удаление строк и слов верификации.
2. Разбиение на потенциальные позиции и цены.
3. Распознавание категории, бренда, модели, памяти, цветов, SIM, EAC.
4. Нормализация через справочники и aliases.
5. Разделение нескольких цветов на отдельные строки.
6. Расчет confidence/status для каждой строки.
7. Блокировка строк с нераспознанным цветом до ручного исправления.
8. Выявление конфликтов, если одинаковый товар указан в одном прайсе с разной ценой.
9. Формирование preview-таблицы.
10. Сохранение только после явного подтверждения.

Строки и слова про верификацию полностью удаляются из рабочего текста до распознавания и не попадают в сохраненные данные.

Если найден конфликт цен внутри одного импорта, система не выбирает цену автоматически. Пользователь вручную выбирает правильную цену в preview.

## Тестовые примеры прайсов

В проекте нужно предусмотреть папку с примерами WhatsApp-прайсов, например:

```text
tests/

    price-lists/
      darkhan/
      talisman/
      f3company/
      mobilagid/
      almaty/
```

Каждый новый формат прайса должен добавляться как fixture и покрываться тестом парсера. Первые реальные примеры ожидаются от Дархана и Талисмана.

## Импорт и обновление каталога

### Частичное обновление

- Создаются или обновляются только распознанные позиции.
- Остальные товары поставщика не меняются.
- Отсутствующие в тексте товары не снимаются с наличия.

### Полная замена прайса

- Новые и существующие предложения обновляются.
- Отсутствующие предложения конкретного поставщика получают статус "Нет в наличии".
- Архивируется и soft-delete помечается предложение конкретного поставщика, а не вся карточка товара.
- Если товар есть у другого поставщика, общая карточка остается активной и не удаляется.
- Через 30 дней архивная запись скрывается и помечается удаленной через soft-delete.
- Перед заменой требуется отдельное подтверждение.

Исходный текст импорта и строки импорта хранятся 12 месяцев. После этого фоновой задачей можно удалить raw text и source rows, сохранив итог импорта, агрегированную статистику и audit log.

## Поставщики и склады

Стартовый глобальный приоритет:

1. Дархан;
2. Талисман;
3. F3Company;
4. Mobilagid;
5. поставщики Алматы по внутреннему порядку.

В первой версии один поставщик связан с одним публичным складом. Для будущего расширения можно оставить возможность связи many-to-many через таблицу `supplier_warehouses`, но в MVP достаточно поля `warehouse_id` в `suppliers`.

Автоматическое переключение на более дешевое предложение запрещено. Если основной поставщик исчез и следующий по приоритету дороже, система меняет основное предложение по наличию, но продажную цену не пересчитывает и создает предупреждение владельцу.

## Ценообразование

Расчет цен должен быть отдельным доменным модулем.

Правила:

- закупочная цена хранится в предложении поставщика;
- утвержденная продажная цена хранится на уровне товара;
- маржа хранится на уровне конкретного товара;
- если маржа не указана, автоматическая маржа не ставится;
- продажная цена не меняется автоматически при изменении закупочной;
- система создает предупреждение и рекомендуемую цену;
- суммы хранятся целыми тенге;
- формульная дробь округляется только до ближайшего целого тенге;
- округления до 500 или 1000 тенге нет.

Банковские проценты:

- Kaspi 24: +18,5%;
- Home 24: +22%;
- Halyk 24: +28%.

Проценты применяются ко всем товарам с утвержденной наличной ценой. Для каждого банка нужен глобальный переключатель включения/отключения.

## Уведомления

Нужны два типа уведомлений:

- внутренние уведомления в системе для сотрудников;
- Telegram-уведомления владельцу или группе.

Telegram-сообщения владельцу могут содержать настоящего поставщика и закупочную цену. Внутренние уведомления продавцу должны использовать только публичный склад и продажные цены.

В первой версии Telegram отправляет сообщения владельцу в личный чат. Возможность группового чата нужно оставить как расширение. Полный Telegram token хранится только в environment variables и не показывается в интерфейсе.

Настраиваемые типы Telegram-уведомлений:

- новый заказ;
- отмена заказа;
- изменение закупочной цены;
- смена основного поставщика;
- ошибка импорта;
- завершение полного импорта.

## Фоновые задачи

Потребуются scheduled jobs:

- ежедневное резервное копирование;
- soft-delete архивных записей старше 30 дней;
- удаление исходных текстов импорта старше 12 месяцев с сохранением итогов;
- проверка прайсов, не обновленных более 24 часов;
- возможно, периодическая агрегация уведомлений;
- очистка старых сессий или технических логов.

На старте это можно реализовать через Vercel Cron или Supabase scheduled jobs.

## Безопасность

Обязательные решения:

- HTTPS-only.
- Supabase Auth с запретом самостоятельной регистрации.
- Вход по логину или email.
- 2FA для владельца и администраторов, с возможностью отключить в тестовой версии.
- Серверная проверка permissions.
- RLS-политики в Postgres.
- Rate limiting для login и API.
- CSRF/XSS защита стандартными средствами Next.js и безопасной работой с формами.
- Валидация всех входных данных на сервере.
- Хранение Telegram token только в переменных окружения.
- Журнал входов и устройств без ограничения по IP и устройствам.
- Автоматический выход после 30 минут бездействия с настройкой срока в интерфейсе.
- Подтверждение критических действий.
- Аудит входов, импортов, цен, ролей, заказов, поставщиков и настроек.

Критические действия для подтверждения:

- полная замена прайса;
- массовое изменение цен;
- удаление или блокировка сотрудника;
- изменение ролей и прав;
- удаление поставщика;
- изменение приоритетов;
- изменение банковских процентов;
- удаление или восстановление архива;
- смена основного предложения;
- критические настройки.

## Производительность

Для каталога:

- серверная пагинация;
- индексы по нормализованным характеристикам товара;
- полнотекстовый или trigram-поиск по модели, бренду, памяти, цвету, SIM и публичному складу;
- отдельные read models/views для каталога;
- lazy loading фотографий;
- кеширование справочников и банковских ставок.

Для нескольких тысяч товаров PostgreSQL и Supabase достаточно, если не отдавать весь каталог целиком.

## Фотографии

Фотографии загружают владелец и администратор с отдельным правом. В первой версии достаточно одной фотографии на модель. На уровне схемы нужно оставить возможность добавить фото для конкретного цвета позже.

Допустимые источники: официальные изображения производителей, изображения поставщиков с разрешением или изображения владельца. Файлы сохраняются в Supabase Storage. Постоянные прямые ссылки на случайные внешние сайты не использовать.

## Экспорт

Экспорт Excel реализуется как серверный action/API, который применяет текущие фильтры и роль пользователя. В первой версии нужен только ручной экспорт.

Поля публичного экспорта: категория, бренд, модель, память, SIM/EAC, цвет, публичный склад, наличная цена, Kaspi 24, Home 24, Halyk 24, дата обновления. Полный экспорт владельца дополнительно включает настоящего поставщика и закупочную цену.

## Деплой и окружения

Использовать Vercel + Supabase. На старте - бесплатный адрес Vercel без собственного домена, но конфигурация должна позволять подключить домен позже.

Окружения:

- `development`;
- `production`;
- Vercel preview deployments для веток и pull requests.

Постоянный staging на бесплатном этапе не нужен.
